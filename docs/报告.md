# 《数据库系统原理》综合实验报告 —— Arena Clash 竞技场对决数据管理平台

> 本文档为实验报告初稿，配合源码一同提交。组内每位成员负责一个功能模块（见"九、分工建议"）。

## 一、选题

### 1.1 系统简介
**Arena Clash 竞技场对决** 是一个面向 MOBA（多人在线竞技）场景的**对战数据管理平台**。它并非游戏客户端，而是围绕"玩家—英雄—对战—赛季—商城—社交"这一业务域构建的数据库信息管理系统，覆盖：

- 玩家注册/登录（JWT 鉴权）、资料与战绩查询；
- 英雄与虚拟商城商品的管理（增删改查）；
- 对战的创建、开始、结算（ELO 评分、MVP 评选）；
- 赛季排行、赛季重置（段位衰减迁移）；
- 好友社交（请求/接受/对比）；
- 数据大屏与审计日志统计。

### 1.2 技术栈
| 层 | 技术 |
|---|---|
| 数据库 | **PostgreSQL 16**（与 OpenGauss 高度兼容）+ Redis 7（实时排行/限流） |
| 后端 | Python 3.13 + FastAPI + SQLAlchemy 2.0（异步、原生 SQL） |
| 前端 | Vue 3 + Vite + Element Plus + ECharts（多页面/多窗体切换） |
| 部署 | Docker Compose 一键编排 |

### 1.3 为什么做该系统
竞技游戏天然产生海量结构化数据（玩家、对战、明细、段位、交易），且业务规则复杂（并发扣款、结算事务、排行计算、级联清理），非常适合综合训练**存储过程、触发器、级联、JOIN、窗口函数、事务与并发控制**等数据库核心技术；同时其管理后台又能完整演示**增、删、改、查、统计**五类基础功能，是一个覆盖面广、可演示性强的选题。

## 二、需求分析

### 2.1 功能需求
1. **数据增删改查（五项齐全）**：管理后台对英雄/商品/玩家执行新增、修改、删除（含级联删除）、查询；各业务页提供统计展示。
2. **存储过程**：对战结算、商品购买（行级锁防超卖）、赛季重置（CTE 段位衰减）、级联删除玩家、接受好友请求、批量建对战等核心业务以存储过程封装，保证原子性。
3. **触发器**：对战状态变化自动同步玩家在线状态；插入对战自动生成编号；删除玩家自动写审计；好友关系增删自动维护好友计数；`updated_at` 自动维护。
4. **级联与 JOIN**：外键级联删除/置空；多表 JOIN 查询（对战详情、排行榜、英雄统计）；player↔player 自连接（好友战绩对比）；子查询/CTE 级联查询。
5. **统计展示**：数据大屏（ECharts）、赛季排行（窗口函数）、审计统计视图。
6. **多窗体切换 + 角色权限**：10 个前端路由页面；管理员/普通玩家按角色显示不同入口。

### 2.2 性能/约束需求
- 并发购买/结算需防止超卖、重复扣款 → 行级锁 `FOR UPDATE`。
- 数据完整性 → CHECK 约束、UNIQUE 约束、生成列、外键策略。
- 高频排行查询 → Redis Sorted Set + 物化视图。

### 2.3 角色
| 角色 | 权限 |
|---|---|
| 游客 | 浏览数据大屏、排行榜、赛季 |
| 玩家 | 以上 + 登录、查询战绩、购买、社交、创建对战 |
| 管理员（`is_admin`） | 以上 + 管理后台（英雄/商品/玩家 CRUD、级联删除）、赛季重置、审计日志 |

## 三、数据库设计

### 3.1 概念设计（E-R 概述）
核心实体与联系（主要外键）：

- **player（玩家）** 1—N **match_detail（对战明细）** N—1 **match_record（对战）** N—1 **season（赛季）**
- **match_detail** N—1 **hero（英雄）**
- **player** 1—N **player_season_rank（玩家赛季段位）** N—1 **rank_tier（段位）**，N—1 **season**
- **player** 1—N **purchase_record（购买记录）** N—1 **shop_item（商品）** N—1 **hero**
- **player** 1—N **audit_log（审计日志）**（操作者，可空）
- **player** N—N **player**（自反联系）：**friendship（好友）** 与 **friend_request（好友请求）**

> E-R 图绘制要点：player 为中心实体；match_detail 是 player×match_record×hero 的联结实体（带属性 kills/deaths/assists/kda…）；friendship/friend_request 表达 player 到 player 的递归多对多。

### 3.2 逻辑设计（关系模式）
共 **13 张表 + 1 物化视图 + 1 普通视图**。关键表：

| 表 | 主要字段 | 说明 |
|---|---|---|
| season | id, name, status, start/end_date | 赛季 |
| rank_tier | id, name, tier_level, min_points | 段位（青铜~黄金 9 级） |
| player | id, username, password_hash, nickname, elo_rating, gold, wins, losses, status, **is_admin**, **friend_count** | 玩家（新增 is_admin/friend_count） |
| hero | id, name, role, price_gold, difficulty, **is_active** | 英雄（新增 is_active 软删标志） |
| match_record | id, match_code(UNIQUE), season_id→season, status, winner_side | 对战 |
| match_detail | id, match_id→match_record, player_id→player, hero_id→hero, kills/deaths/assists, **kda(生成列)**, is_mvp, extra_data(JSONB) | 对战明细 |
| player_season_rank | id, player_id→player, season_id, rank_tier_id, rank_points, **win_rate(生成列)** | 玩家赛季段位 |
| shop_item | id, name, item_type, hero_id→hero, price_gold, stock, is_limited | 商品 |
| purchase_record | id, player_id→player, item_id→shop_item, price_paid, UNIQUE(player_id,item_id) | 购买记录 |
| audit_log | id, player_id→player, action, detail(JSONB) | 审计日志 |
| **friend_request**（新增） | id, requester_id→player, addressee_id→player, status | 好友请求 |
| **friendship**（新增） | id, player_id→player, friend_id→player, UNIQUE(player_id,friend_id) | 好友关系（双向各一行） |
| **mv_season_statistics** | 赛季×英雄 的 picks/wins/win_rate/avg_kda/mvp_count | 物化视图 |
| **v_audit_action_stats**（新增） | action, cnt, last_at, last_7d | 审计统计视图 |

**外键级联策略**（本次重点新增，删一个玩家即可演示多表级联）：

| 外键 | ON DELETE | 理由 |
|---|---|---|
| match_detail.match_id → match_record | CASCADE | 明细属于对战 |
| match_detail.player_id → player | CASCADE | 删玩家清其参战记录 |
| match_detail.hero_id → hero | RESTRICT（+软删） | 英雄被引用禁止硬删，改 is_active=false |
| player_season_rank.player_id → player | CASCADE | 段位属于玩家 |
| shop_item.hero_id → hero | SET NULL | 英雄删除后商品解除关联 |
| purchase_record.player_id / item_id | CASCADE | 购买记录属于玩家/商品 |
| audit_log.player_id → player | SET NULL | 删玩家保留审计，仅置空操作者 |
| friendship/friend_request.*_id → player | CASCADE | 删玩家清其社交关系 |

### 3.3 物理设计（索引/存储）
- **B-Tree 索引**：username、elo_rating DESC、status；match(season_id,status)、created_at DESC、match_code；detail(match_id)、(player_id,created_at DESC)、(hero_id)；psr(season_id,rank_points DESC)；friend_request(addressee_id,status)、friendship(player_id) 等。
- **部分索引**：`idx_detail_mvp ... WHERE is_mvp=TRUE`、`idx_shop_limited ... WHERE is_limited=TRUE`。
- **GIN 索引**：match_detail.extra_data、audit_log.detail（JSONB 检索）。
- **唯一索引**：mv_season_statistics(season_id,hero_id)（支撑 REFRESH）。
- **生成列（STORED）**：match_detail.kda、player_season_rank.win_rate。

## 四、详细设计

### 4.1 存储过程（4 个原有 + 4 个新增）
| 存储过程 | 功能 | 关键技术 |
|---|---|---|
| fn_settle_match | 对战结算：经验/金币/ELO、评 MVP | 行级锁、循环、CHECK 兼容 |
| fn_purchase_item | 购买商品 | 行级锁防超卖、多重校验 |
| fn_reset_season | 赛季重置 | **CTE** 段位衰减批量迁移 |
| fn_generate_match_code | 生成唯一对战编号 | 循环去重 |
| **fn_delete_player_cascade**（新） | 级联删除玩家 + 审计 + 刷新物化视图 | 行级锁、级联、管理员保护 |
| **fn_accept_friend_request**（新） | 接受好友：双向插入 friendship | 事务、行级锁、ON CONFLICT |
| **fn_create_match_with_players**（新） | 用 JSONB 数组一次性建满员对战 | jsonb_array_elements 驱动 |
| **fn_soft_delete_hero**（新） | 英雄软删 + 解除商品关联 | 替代硬删，保护历史 |

### 4.2 触发器（3 个原有 + 2 个新增）
| 触发器 | 时机 | 功能 |
|---|---|---|
| trg_match_status_change | AFTER UPDATE OF status | 对战开始/结束 → 同步玩家 in_match/online |
| trg_auto_match_code | BEFORE INSERT | 自动生成 match_code |
| trg_*_updated | BEFORE UPDATE | 维护 updated_at |
| **trg_audit_player_delete**（新） | **AFTER DELETE ON player** | 任何方式删玩家都写审计（此前无 DELETE 触发器） |
| **trg_friendship_count**（新） | AFTER INSERT/DELETE ON friendship | 维护 player.friend_count 反范式计数 |

### 4.3 典型查询
- **多表 JOIN**：对战详情 `match_detail ⋈ player ⋈ hero ⋈ match_record`；排行榜 `player_season_rank ⋈ player ⋈ rank_tier`。
- **窗口函数**：`RANK()/LAG()/LEAD()`（排行分差）、`NTILE(4)`（段位分桶）、`AVG() OVER (ROWS BETWEEN 4 PRECEDING AND CURRENT ROW)`（近 5 场滑动 KDA）。
- **自连接**：`player me JOIN player fr ON fr.id=:friend`（好友战绩对比）。
- **子查询/级联查询**：赛季列表中"参与人数"子查询；fn_reset_season 中 CTE `old_ranks → resolved → INSERT`。
- **聚合统计**：物化视图按赛季×英雄聚合；`v_audit_action_stats` 按操作类型 `COUNT(*) FILTER (...)`。

## 五、系统实现

### 5.1 多窗体（10 个前端路由）
数据大屏 / 排行榜 / 玩家 / 对战 / 商城 / **对战中心** / **社交** / **赛季中心** / **管理后台(仅管理员)** / **审计日志(仅管理员)**。导航按 `is_admin` 条件渲染，并配 `router.beforeEach` 守卫。

### 5.2 五个新增模块（每人一个）
1. **管理后台 AdminConsole**：英雄/商品/玩家全 CRUD；玩家删除调 `fn_delete_player_cascade` 演示级联；英雄硬删被 RESTRICT 拦截后弹窗改"下架"；一键刷新物化视图。
2. **对战中心 MatchCenter**：建对战（`fn_create_match_with_players`）→ 开始（触发器把玩家置 in_match）→ 结算（`fn_settle_match` 写 KDA/MVP/ELO）。
3. **社交 Social**：好友请求/接受(`fn_accept_friend_request`)/拒绝、好友列表、删除、战绩对比（自连接 + ECharts）。
4. **赛季中心 SeasonCenter**：赛季列表、重置(`fn_reset_season`)、历史排名（RANK/LAG/LEAD）。
5. **审计日志 AuditLog**：按操作类型/玩家过滤查询 JSONB 日志，统计视图卡片 + ECharts 饼图。

### 5.3 鉴权与角色
登录签发的 JWT 含 `is_admin`；后端 `require_admin` 依赖**回查数据库** `player.is_admin`（避免提权后旧 token 失效问题），管理类端点全部加该依赖。

### 5.4 部署
`docker compose up -d` 启动 postgres/redis/backend/frontend；`sql/*.sql` 以 `01-init→04-sample_data` 顺序在空数据卷初始化。前端 3000、后端 8000、API 文档 `/docs`。

## 七、存在的问题及思路
1. **玩家硬级联删除会改变历史聚合**：删除已完成对战的参与者会使该场次统计（物化视图、英雄胜率）发生变化，甚至出现不足 10 人的对战。思路：生产环境可对玩家也采用软删（`is_active`），仅对测试账号硬删；本实验为直观演示"级联删除"而采用硬删，并在删除后刷新物化视图。
2. **物化视图刷新成本**：`REFRESH MATERIALIZED VIEW` 取 ACCESS EXCLUSIVE 锁，数据量大时有阻塞。已修复：`fn_delete_player_cascade` 与 `/api/admin/refresh-stats` 均已改为 `REFRESH MATERIALIZED VIEW CONCURRENTLY`，利用已建的唯一索引进行无锁刷新，不阻塞读取。
3. **旧 JWT 与角色变更**：`require_admin` 回查数据库而非只读 token，避免提权/降权后 token 失效；代价是每次多一次查询。
4. **OpenGauss 兼容性**：本系统 SQL 绝大部分为标准 SQL/PL-pgSQL，迁移到 OpenGauss 时需重点复验：物化视图 `REFRESH`、`GENERATED ALWAYS AS ... STORED` 生成列、JSONB/GIN 索引、`FILTER` 子句。`sql/migrations.sql` 提供了对已有库的幂等升级脚本。

## 八、总结
本系统在原有"对战/排行/商城/大屏"的基础上，补齐了课程要求的**完整增删改查（尤其是删除与级联）**、新增 **4 个存储过程 / 2 个触发器 / 1 张统计视图 / 2 张社交表 / 一套外键级联策略**，并通过 **5 个新窗体**把存储过程与触发器"跑给评审看"。系统完整覆盖了存储过程、触发器、级联查询、JOIN、窗口函数、事务与并发控制、统计展示、多窗体切换与角色权限等数据库核心知识点，达到了综合实验的训练目标。
